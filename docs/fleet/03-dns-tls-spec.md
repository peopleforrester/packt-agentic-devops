# 03 · DNS and TLS specification

Goal: every student opens `https://studentN.packt.ai-enhanced-devops.com` with a valid certificate and a
`wss://` terminal socket. No Chrome "Not secure", no raw ELB hostnames.

## Why per-cluster certificates are impossible

Let's Encrypt limits **certificates per registered domain to 50 per 7 days**, counted at the eTLD+1, so
everything under `ai-enhanced-devops.com` shares one bucket. 250 per-host certs would breach it by 5×.
This is not a tuning problem; it is a hard wall, and it is why the design below uses exactly **one**
wildcard certificate.

It is also why Unleashed's 250 attendees ended up on plain `http://<raw-elb>` — the wildcard router was
built and worked, but was only wired for instructor clusters. We are turning it on for everyone.

## Architecture

```
browser  https://student42.packt.ai-enhanced-devops.com/
   |
   |  (1) DNS: *.packt.ai-enhanced-devops.com  CNAME -> Railway edge
   |          _acme-challenge.packt...          CNAME -> Railway   (delegated; Railway does DNS-01)
   |          + ownership TXT
   v
(2) RAILWAY EDGE  -- TLS terminates here, ONE wildcard cert, auto-renewed. We run no ACME client.
   |  plain HTTP, Host preserved
   v
(3) CADDY router (own Railway service, `auto_https off`)
       map {host} {upstream} from routes.map
       no match -> friendly 404 page
   |  http://<cluster ELB>:80
   v
(4) per-cluster VTT LoadBalancer
   v
(5) nginx console -> /terminal/ -> ttyd (websocket upgrade)
```

TLS is edge-only. Everything inward is plain HTTP on a disposable cluster that lives hours.

## Why Railway issues the cert, not us

Railway supports wildcard custom domains and delegates `_acme-challenge` to its own side. That sidesteps
the Namecheap API IP-allowlist entirely, because we never run DNS-01 ourselves. Adding the wildcard
domain yields **two CNAMEs (wildcard + `_acme-challenge`) plus one ownership TXT — all three required**
or the cert will not issue.

Hard rule: **never delete and recreate the Railway service** that holds the domain. Re-adding forces new
LE validation and new records.

## DNS records (Namecheap)

Exactly three static records, set once, never per cluster:

| Host | Type | Target |
|---|---|---|
| `*.packt` | CNAME | Railway edge target |
| `_acme-challenge.packt` | CNAME | Railway-provided |
| `packt` (existing) | CNAME | Railway (claim app) — already live |

Two Namecheap facts that will bite otherwise:

1. **`setHosts` replaces the ENTIRE zone.** Any writer must read the existing records first and merge, or
   it silently deletes the claim app's records, the email records (SES/Resend MX, SPF, DKIM, DMARC), and
   everything else. The existing `infra/dns/set-demo-dns.py` in the Unleashed repo already implements
   read-then-merge and is the model to copy.
2. **The API checks `ClientIp` against an allowlist.** netcup (`152.53.192.39`) is whitelisted, so DNS
   work runs from there. Propagation can take up to ~60 minutes — do the records early, not at the gate.

A wildcard matches **exactly one label**, so hostnames must be single-label under `packt`: `student42`
is fine, `42.student` is not.

## Router map

`routes.map`, one line per cluster:

```
student42.packt.ai-enhanced-devops.com  <cluster-elb-hostname>:80
```

Generated by `fleet.sh routes` from the **live** cluster LBs (never hand-edited), committed and pushed,
which triggers the Railway rebuild. Do not put `[skip ci]` on that commit — it suppresses the redeploy
and leaves the map stale, which is a silent outage for every student.

Unknown host falls through to a friendly 404, which is the correct behaviour for a cluster that was
reaped or not yet provisioned.

Scale note: 250 map lines is fine (Caddy's `map` is a hash lookup) but the router is **one replica in the
path for all 250 students**. Size it deliberately, and keep the raw ELB hostname in the pool as a
documented fallback.

## Why `wss://` comes free

The terminal is a **same-origin iframe** (`<iframe src="/terminal/">`) under the same hostname as the lab
page, so ttyd's client derives its scheme from `location.protocol`. Serve the hostname over HTTPS and the
socket becomes `wss://` with no code change and no mixed-content risk. This is the entire reason the VTT
is path-routed under one host rather than split across hostnames — keep it that way.

In-cluster nginx already carries the upgrade headers and a 3600s read timeout so an idle shell survives a
four-hour workshop.

## Pool interaction

Once the router is live, `terminal_url` in the pool becomes
`https://studentN.packt.ai-enhanced-devops.com/?cluster=studentN` instead of the raw ELB URL. Keep the
raw hostname in a spare column as the fallback.

## Verification

Owned by `tests/test_tls.sh` (L4 in `04-verification-tests.md`): chain validates, SAN covers the host,
expiry beyond the event, HTTP redirects to HTTPS, and the websocket upgrades with a 101. Written and
failing **before** the router exists.

## Teardown

There is none for DNS, by design. The wildcard record and cert are static and permanent; nothing
per-cluster ever enters DNS. A stale `routes.map` line self-heals on the next `routes` run and 404s
harmlessly in the meantime.
